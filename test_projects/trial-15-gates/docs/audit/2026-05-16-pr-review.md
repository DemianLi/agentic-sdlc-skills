## Findings

### FIND-1 — MEDIUM: Loop-based character filter is less idiomatic than regex

**File**: `src/slugify/__init__.py`, lines 35–47
**Description**: The character filter uses two explicit character-by-character loops instead of a single `re.sub()` pass. While functionally correct and readable, the loop approach is ~3× slower for large inputs than a compiled regex.

**Current code**:
```python
result = []
for char in text:
    if char.isspace():
        result.append(separator)
    else:
        result.append(char)
text = "".join(result)

result = []
for char in text:
    if (char >= 'a' and char <= 'z') or (char >= '0' and char <= '9') or char == separator:
        result.append(char)
return "".join(result)
```

**Suggested alternative** (v1.1):
```python
import re
text = text.lower()
text = re.sub(r'\s', separator, text)
return re.sub(r'[^a-z0-9' + re.escape(separator) + r']', '', text)
```

**Decision**: Defer to v1.1. Current performance (1.0ms per 10K-char input, SLO < 500ms) is well within bounds. Loop approach has zero external dependencies.

### FIND-2 — LOW: `__main__.py` strips trailing `\n` before slugifying — may silently swallow `\r\n` on Windows

**File**: `src/slugify/__main__.py`, line 34–35
**Description**: `text = text[:-1]` removes exactly one character only when `text.endswith("\n")`. On Windows, `sys.stdin.read()` may return `\r\n` line endings, so the trailing `\r` would become a separator. This is only a concern if slugify is run on Windows against Windows-style line-ending stdin.

**Impact**: Low — target platform is POSIX. Out of scope for v1.0 per requirements (no Windows mention).

**Decision**: Acceptable for v1.0. Add `text = text.rstrip('\r\n')` in v1.1 for cross-platform robustness.

---

### FIND-3 — LOW: No validation of `separator` argument length

**File**: `src/slugify/__init__.py` and `src/slugify/__main__.py`
**Description**: Design document DC-1 states separator must be a single character, but neither the library function nor the CLI validates this. Passing `separator="--"` (two hyphens) produces unexpected but not erroneous results — multi-char separator is preserved through the filter.

**Impact**: Low — misuse by the caller, not a bug. The library contract (DC-1: single character) is documented.

**Decision**: Acceptable for v1.0. Add `assert len(separator) == 1` or raise `ValueError` in v1.1.
