## Findings

### FIND-1 [MEDIUM]: Duplicated fence-tracking logic

**Files**: `core.py:19-39` (`parse_headers`) and `core.py:61-83` (`_find_markers_outside_fences`)

Both functions contain nearly identical code-fence tracking loops (`in_fence`, `fence_char`, `_FENCE_RE.match`). If the fence-tracking behavior changes (e.g., handling indented fences), both copies must be updated. Violation of DRY.

**Recommendation**: Extract a `_lines_outside_fences(text: str) -> Iterator[tuple[int, str]]` generator that yields `(line_index, line)` for lines not inside a code fence. Both `parse_headers` and `_find_markers_outside_fences` consume it.

**Impact**: Not a bug today. Maintenance risk as fence rules evolve. Defer to v1.1.

---

### FIND-2 [LOW]: Misleading variable name `stripped_lines` in `insert_toc`

**File**: `core.py:90`

```python
stripped_lines = [l.rstrip("\n") for l in lines]
```

`stripped_lines` implies full whitespace stripping, but only trailing `\n` is removed. Name suggests more aggressive stripping than occurs.

**Recommendation**: Rename to `lines_no_newline` or `line_texts` to clarify intent.

**Impact**: Cosmetic. No behavior change.

---

### FIND-3 [LOW]: `insert_toc` trailing-newline stripping is overly aggressive

**File**: `core.py:104-105`

```python
if not text.endswith("\n") and result.endswith("\n"):
    result = result.rstrip("\n")
```

`rstrip("\n")` removes ALL trailing newlines. If the original document body ends with multiple newlines (unusual but valid), this silently collapses them to zero. The intent is to not ADD a newline that wasn't there, not to remove existing ones.

**Recommendation**: Strip exactly one trailing `\n` if it was added by the TOC block:

```python
if not text.endswith("\n") and result.endswith("\n"):
    result = result[:-1]
```

**Impact**: Edge case. Only affects documents that don't end with `\n` (atypical). Tests pass either way because test inputs don't end with `\n`.
