# QA — Agentic SDLC 開發階段清單

> 此清單為本 repo 所有 skill 的設計意圖基準。評估 skill 的對齊度時，以此文件各子步驟描述為 ground truth。

---

## 零、獨立工具階段 (Standalone Skills)

> s0 系列為流水線外的獨立工具，可在任意階段使用，不佔流水線卡點。

**0.1 發散探索 (Brainstorm)**：收集初始想法、問題陳述或技術路線假設，輸出結構化的思路發散報告。對應 skill：`s0-brainstorm`。

**0.2 功能溯源 (Trace Feature)**：從任意入口（UI 元件、API endpoint、資料表欄位）反向追溯功能的完整實作鏈，輸出跨層追蹤圖。對應 skill：`s0-trace-feature`。

**0.3 Skill 結構審計 (Eval Skill)**：對單一 SKILL.md 執行 6 項生產就緒標準評分（衝突防禦、雙向阻斷、輸入清洗、漸進披露、優雅降級、漂移監控），輸出診斷報告。對應 skill：`s0-eval-skill`。

**0.4 對齊度批次掃描 (Eval Alignment)**：批次掃描所有 s1–s7 skill，比對設計意圖與現行內容的語意落差，輸出 ALIGNED / PARTIAL / DRIFTED 狀態表。對應 skill：`s0-eval-alignment`。

**0.5 Token 預算審計 (Skill Budget)**：對新增或修改的 SKILL.md 執行三軸（D 描述精確度、I 索引覆蓋率、S 檔案大小）token 效率審計，確保 deferred loading 命中率與加載成本維持在預算內。對應 skill：`s0-skill-budget`。

**0.6 語意內容驗證 (Semantic Validate)**：以 `json_query` / `regex_match` DSL 驗證工件的實際內容（而非僅確認檔案存在），作為 HARD-GATE 的補充驗證層，確保工件語意符合前置條件要求。對應 skill：`s0-semantic-validate`。

## 一、初始化與根本規則定義階段 (Initialization & Base Rules)

**1.1 項目根本規則定義**：確立編碼規範（Lint 規則）、系統架構範式（如 Clean Architecture）、安全性與合規性底線。

**1.2 基礎上下文配置**：建立專案全域 Context（如 .claudecode/ 或 Rules.md），配置 AI 代理的技能邊界與工具權限。

**1.3 技術棧與依賴鎖定**：明確定案開發語言版本、框架、核心第三方套件與資料庫選型。

---

## 二、需求對齊與結構化階段 (Requirement Alignment)

**2.1 原始構思捕獲**：收集業務方的初始想法、痛點描述或模糊的市場需求。

**2.2 需求對齊與衝突消除**：透過問答澄清模糊地帶，定義系統邊界，確保開發團隊（或 Agents）與業務方認知完全一致。

**2.3 需求結構化描述**：將對齊後的需求轉化為標準化、無歧義的文檔（如 PRD、標準 User Story、Gherkin 語法的 BDD 測試案例）。

**2.4 文檔沉澱與快照**：將新需求寫入知識庫，生成當前迭代的需求上下文快照（Context Snapshot），防止後續開發偏離目標。

---

## 三、技術方案制定與原子化拆解階段 (Architecture & Breakdown)

**3.1 現有系統評估**：分析新需求對現有系統代碼、資料庫 Schema 及現有 API 的影響。

**3.2 技術方案設計 (Design Doc)**：制定具體技術實現路徑，設計資料結構、類別圖、接口合約（API Contract）與系統時序圖。

**3.3 原子化任務拆解 (WBS)**：將技術方案解構成相互獨立、無狀態依賴或依賴清晰的「最小可執行單元」（Atomic Tasks）。

**3.4 依賴拓撲建立 (DAG)**：梳理原子任務的先後順序，建立有向無環圖（DAG），鎖定並發開發與交付的順序。

---

## 四、開發實現與本地調試階段 (Implementation & Local Debug)

**4.1 研發環境就緒**：拉取正確的 Git 工作區（Worktree 或 Feature Branch），初始化開發沙盒。

**4.2 原子任務並發實現**：依照第一階段的「根本規則」與第三階段的「原子任務說明」，專注編寫核心業務邏輯代碼。

**4.3 單元測試同步編寫 (TDD/Test-Aside)**：在編寫業務代碼的同時，編寫對應的單元測試（Unit Test），確保代碼具備可測性。

**4.4 本地編譯與微調試**：進行本地編譯，捕獲語法與邏輯錯誤，分析日誌與堆疊追蹤（Stack Trace），在本地環境完成初步調試。

---

## 五、代碼審查與靜態驗證階段 (Code Review & Verification)

**5.1 靜態代碼分析 (Lint/SAST)**：自動化運行 Lint、格式化工具與安全性掃描，排查潛在的漏洞與代碼壞味道（Code Smells）。

**5.2 根本規則合規審查**：審查代碼是否嚴格符合第一階段定義的架構範式與設計準則（如是否破壞了解耦原則）。

**5.3 代碼評審與意見反饋 (PR Review)**：進行同行評審，提出重構、性能優化或邏輯漏洞的修改意見。

**5.4 排障與結構優化**：根據評審意見快速迭代，修復問題並對代碼進行局部的重構優化。

---

## 六、動態測試與動態驗證階段 (Dynamic Testing)

**6.1 自動化整合測試**：將多個原子任務的代碼合併後，執行模組間的整合測試（Integration Test），驗證接口調用。

**6.2 端到端與邊界驗證 (E2E Test)**：運行自動化 E2E 測試，模擬真實用戶行為，嚴格對齊第二階段結構化需求中的邊界條件。

**6.3 效能與壓力測試**：驗證高並發、大數據量下的系統表現（記憶體洩漏、資料庫死鎖、響應時間等）。

**6.4 結果最終驗證**：產出自動化測試報告與代碼覆蓋率（Coverage）報告，確認結果完全符合預期，發出准予交付信號。

---

## 七、交付發佈與閉環迭代階段 (Delivery & Iteration)

**7.1 構建與封裝 (Artifact Creation)**：執行 CI/CD 流程，將通過驗證的代碼編譯、打包成可部署的鏡像或發佈檔。

**7.2 變更日誌沉澱 (Release Notes)**：自動化比對 Git 提交記錄，生成本次迭代的變更日誌、升級指南與 API 變更文檔。

**7.3 生產環境部署 (Deployment)**：透過 GitOps 或部署工具將產物安全推送至預發布/生產環境。

**7.4 運維監控與反饋閉環**：開啟遙測（Telemetry）與日誌監控，捕獲生產環境異常，並將收集到的新問題或新優化點，作為下一個迭代循環的「初始構思」輸入。
